# 播放时 CPU <1% 事件驱动调度设计

日期：2026-04-04

## 1. 背景

当前 `wallpaper_app.exe` 在播放视频时，主循环与解码泵均采用高频定时轮询，并在 60fps 场景下启用 `timeBeginPeriod(1)`。这会显著提升系统与进程唤醒频率，导致任务管理器“进程”页 CPU 难以稳定低于 1%。

用户目标明确：

- 任务管理器“进程”页 `wallpaper_app.exe` CPU < 1%，以稳定播放后 60 秒平均值为准。
- 不降低画质：不降分辨率、不改采样/色彩、不降低码率/清晰度。
- 允许渲染上限跟随视频源帧率（例如 30fps 视频以 30fps 输出）。
- 不要求向下兼容旧配置格式。

## 2. 目标

- 将播放时 CPU 控制在 < 1%（任务管理器进程页口径，60 秒平均）。
- 保持画质与输出分辨率，不做下采样与码率调整。
- 将渲染节拍与源帧率对齐，避免无意义重复 present。
- 用事件驱动替代主循环与解码泵的高频轮询。

## 3. 非目标

- 不保证在所有硬件/所有 60fps CPU 软解场景下必然 < 1%。
- 不引入跨平台抽象或兼容路径。
- 不增加新的 UI 或交互功能。

## 4. 约束与原则

- **No backward compatibility**：允许破坏旧配置格式，缺失字段直接用默认值重建。
- **以高性能为主**：消除无效唤醒，降低 CPU 空转。
- **复杂逻辑必注释**：事件等待与唤醒路径需要明确说明原因与边界。
- **优先异步**：能等待事件就不轮询。

## 5. 方案概述（推荐）

核心思路：构建“事件驱动的等待集合（Wait Set）”，把“下一帧渲染”“解码就绪”“探测/指标周期”全部变成可等待事件或定时器，主线程与解码线程只在必要时被唤醒。

### 5.1 主线程等待模型（Main Loop Wait Set）

- 使用 `MsgWaitForMultipleObjectsEx` 等待：
  - Windows 消息队列
  - 解码帧就绪事件（如可用）
  - 下一次渲染时刻的 Waitable Timer
  - 下一次探测/指标采样时刻的 Waitable Timer
- 仅在事件触发时唤醒，避免固定 `Sleep(2~40ms)` 轮询。
- `timeBeginPeriod(1)` 只在必须高精度且无法用 waitable timer 精确等待时启用，常态关闭。

### 5.2 渲染节拍策略

- 渲染上限：`renderCap = min(config.fpsCap, sourceFps)`，`sourceFps` 不可得时回退为 `config.fpsCap`。
- 渲染触发条件：
  - 首帧渲染
  - 解码序列号提升（存在新帧）
- 这样避免重复 present 导致的 CPU 消耗，而不降低画质或分辨率。

### 5.3 解码泵等待策略

- 如果 `SupportsFrameReadyNotifier()` 为 true：
  - 解码线程等待“帧就绪事件”，只在新帧到达时取帧。
- 如果不支持 notifier：
  - 使用 Waitable Timer 以 `sourceFps` 节拍唤醒，取最新帧。
- 彻底移除 `2ms` 轮询退避路径，解码泵唤醒频率 <= 源帧率。

### 5.4 可选 DXGI Frame Latency Waitable Object

- 保留当前默认关闭策略。
- 新增 `frameLatencyWaitableMode`（默认 `off`，允许 `auto`）。
  - `auto`：创建成功且稳定时启用，否则自动回退。
  - 仍保留现有 “兼容性风险” 注释与回退逻辑。

## 6. 配置变更（破坏性可接受）

新增配置字段：

- `renderCapMode`：`source | fixed`，默认 `source`
- `frameLatencyWaitableMode`：`off | auto`，默认 `off`

缺失字段按默认值补齐，不做旧格式兼容分支。

## 7. 关键模块与职责

- `App`：持有 Wait Set，驱动主循环、探测与渲染节拍。
- `RenderScheduler`：新增“按源帧率更新节拍”的入口（仅调整节拍，不修改画质）。
- `DecodePipeline`：提供帧就绪事件（如可用），否则由解码泵定时器驱动。
- `LoopSleepPolicy`：替换为“事件驱动节拍策略”，保留必要的边界 clamp 逻辑。

## 8. 数据流（简述）

1. 解码线程等待帧就绪事件或定时器。
2. 取到新帧后更新序列号并触发“渲染事件/标志”。
3. 主线程 Wait Set 被唤醒，确认有新帧后执行 present。
4. 探测/指标采样按定时器触发，不再每帧轮询。

## 9. 失败与回退策略

- 事件创建失败：回退到当前 Sleep 路径，但记录一次性告警。
- 帧就绪事件不可用：自动切换为“按源帧率定时器唤醒”。
- `frameLatencyWaitableObject` 失败或出现异常：强制关闭并回退。

## 10. 测试与验证

### 10.1 功能验证

- 1080p30 H264：渲染帧率跟随 30fps，画质不变。
- 1080p60 H264：保持 60fps，不降分辨率，不降采样。

### 10.2 性能验证

- 任务管理器“进程”页观察 `wallpaper_app.exe` CPU：稳定播放 60 秒平均 < 1%。
- 记录 `metrics.csv` 中 CPU 指标用于辅助验证。

## 11. 风险与取舍

- 事件驱动后主循环结构更复杂，需补充注释与边界说明。
- 低频唤醒可能暴露隐藏的状态同步问题，需要补充测试。
- 60fps CPU 软解极端场景可能仍 >1%，需要明确“硬解优先”的现实边界。

## 12. 分阶段实施建议

Phase 1：引入 Main Loop Wait Set + 渲染按源帧率节拍

- 主循环事件化
- 渲染节拍与源帧率绑定
- 替换原有 sleep 轮询

Phase 2：解码泵事件化

- 支持帧就绪事件优先
- fallback 模式使用 waitable timer

Phase 3：可选 DXGI waitable object

- `frameLatencyWaitableMode=auto` 实现与回退

## 13. 成功判定

- 1080p30/1080p60 视频播放时：CPU < 1%（任务管理器进程页，60 秒平均）。
- 画质与分辨率保持不变。
- 不引入明显帧丢失或播放抖动。
